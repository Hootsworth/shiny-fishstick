Documentation

Welcome to the Shiny Fishstick reference documentation. Learn how to compile arbitrary web platforms into structural, high-speed API layers for AI browser agents.

Note: Make sure you have Node.js 18+ and Python 3.9+ installed in your developer workspace to compile targets.

Overview

AI browser agents spend up to 90% of their runtime downloading raw HTML resources, rendering CSS styles, guessing dynamic identifiers, and waiting for heavy scripts to execute. Shiny Fishstick acts as a pre-compiler mapping these heavy interactions into clean SDK bindings.

Compilation Pipeline

The compiler runs target websites through a multi-stage compilation framework:

  1. Crawling & Mapping: Playwright navigates seed pages, clusters dynamic parameters, and tracks state rules.
  2. Scoring Selectors: Gathers potential interactive controls and rates their uniqueness weights.
  3. Interception & Upgrades: Monitored fetches replace button clicks with REST queries.
  4. FSM Compilation: Chains consecutive actions into transition nodes.

BFS Crawler

The Playwright-backed Crawler traversal scans targets starting at a seed URL. To prevent path expansion loops on database items (such as `/item/1` or `/item/52`), it clusters path routes into structural parameters, like `/item/{id}`.

# Start the crawler backend scan
python3 -m backend.app.main --crawl http://localhost:8001

Heuristic Selector Analyzer

To withstand layout edits and CSS drift (common in Tailwind compiles), target DOM nodes are ranked by specificity and intent:

  • data-testid (Score: 1.0) — Purpose-built QA keys.
  • id (Score: 0.9) — Unique element keys.
  • name (Score: 0.7) — Form inputs.
  • class (Score: 0.4) — Visual identifiers.
  • XPath / Tag (Score: 0.1) — Layout structure.

Intent & AI Engine

Clusters separated form elements (like email fields, password fields, and submit buttons) into singular, high-level methods (e.g. login(email, password)). The engine uses the Gemini API when available, and falls back to local regex-based classification lists offline.

API Intercept & Upgrades

When the crawler acts on a selector, the interceptor snoops network sockets. If a button trigger fires an XHR fetch query, the click is mapped directly to a REST api method in the final SDK class, completely bypassing page layouts during runtime.

Session Serialization (Feature A)

If the crawler hits a validation wall, it runs credentials matching. Upon success, the session snapshots all active tokens and databases:

  • HTTP Only & Session Cookies
  • LocalStorage keys
  • SessionStorage keys

The generated SDK allows you to save and seed these credentials using standard serializations:

# Save authorized states
session_data = sdk.export_session()

# Bootstrap new clients without logging in again
sdk_new = ShinyFishstickSiteSDK()
sdk_new.start(session_data=session_data)

Token Propagation (Feature C)

For API upgraded endpoints, the SDK automatically correlates intercepted headers (like Authorization: Bearer [JWT]) with their storage source (cookies or localStorage keys), propagating authentication variables back into API actions dynamically.

Model Context Protocol (MCP) (Feature D)

The compiler generates a Model Context Protocol server. This lets LLMs running in workspaces (like Cursor or Claude Desktop) invoke site routines directly as standard tools.

Setup cursor tool config:

# Run in a background terminal
python3 mcp_server.py

Add it to Cursor Settings under "MCP Servers" by naming it and passing the command: python3 /absolute/path/to/mcp_server.py

E2E SDK Test Generation (Feature E)

Spits out an integration verification suite (test_sdk.py) mapping the exact workflow states discovered. Run it to ensure that target selectors haven't drifted:

# Runs generated integration assertions
python3 test_sdk.py
